Data Pipeline Orchestration

Plugin Architecture

SPI-based extension points Each plugin type has a protobuf contract Hot-swappable at deploy time

Pipeline Execution Flow

Dry-run planning before execution Transactional commit/rollback Runtime-agnostic task submission

Strengths

  • Extreme extensibility via plugin SPI
  • Type safety through protobuf contracts
  • Dry-run / plan-first execution model
  • Clean separation of compute and storage

Shortcomings

  • Plugin abstraction adds onboarding overhead
  • Single-process boundary limits horizontal scaling
  • Polling-based status checks add latency
  • Retries left to plugin implementations

Pub/Sub: Event Processing Framework

Event Flow

Fully decoupled producers Serverless auto-scaling Audit trail to BigQuery

Handler Architecture

Template method pattern Per-instance webhook caching Extensible handler registration

Strengths

  • Fully decoupled producers and consumers
  • Serverless auto-scaling
  • Multi-tenant by design
  • Extensible handler registration

Shortcomings

  • At-least-once delivery requires idempotent consumers
  • No ordering guarantees across partitions
  • Webhook fan-out adds tail latency
  • Centralized handler subject to noisy-neighbor issues
